# PROJECT ARCHITECTURE & CODING RULES

This document defines the mandatory engineering standards for this project.
All code must follow these rules to ensure maintainability, scalability, and production quality.

---

1. CORE PRINCIPLES

---

All development must follow these principles:

* DRY (Don't Repeat Yourself)
* SOLID Principles
* Clean Architecture
* Separation of Concerns
* High Performance
* Security First
* Scalability
* Maintainability
* Testability

Avoid quick fixes or shortcuts that violate these principles.

---

2. TECHNOLOGY STACK

---

Backend Framework:

* ASP.NET Core MVC (.NET 6)

Database:

* SQL Server

ORM:

* Entity Framework Core

Authentication:

* JWT (JSON Web Token)

Architecture Pattern:

* 3-Layer Architecture

  Controllers  →  Services  →  Repositories

---

3. ARCHITECTURE RULES

---

The system must follow strict separation between layers.

Controllers

* Responsible only for HTTP handling
* Input validation
* Returning HTTP responses
* No business logic allowed

Services

* Contain ALL business logic
* Coordinate multiple repositories
* Handle application rules

Repositories

* Handle data access only
* Communicate with DbContext
* No business logic

Data must flow strictly:

Controller → Service → Repository → Database

Never bypass layers.

---

4. CLEAN ARCHITECTURE FOLDER STRUCTURE

---

The project must be organized in a clear layered structure.

Example:

/Controllers
/Services
/Interfaces
/Implementations
/Repositories
/Interfaces
/Implementations
/Entities
/DTOs
/Mappings
/Validators
/Middleware
/Data
/Configurations
/Migrations
/Common
/Helpers

Responsibilities must be clearly separated.

---

5. ENTITY FRAMEWORK CORE RULES

---

ORM:

* Entity Framework Core must be used for database access.

Model Configuration:

* Use Fluent API for relationships and constraints.
* Do NOT use Data Annotations for navigation configuration.

DbContext:

* All entity configuration must be inside `OnModelCreating`.
* Use separate configuration classes when models grow large.

Indexes:

* Define indexes for frequently queried fields.

Migrations:

* Database must be migration-driven.
* Never manually modify production schema without migrations.

---

6. DTO USAGE

---

DTOs must be used between layers.

Rules:

* Controllers never expose Entities directly.
* Services return DTOs.
* Requests use dedicated DTOs.

Example types:

CreateXDto
UpdateXDto
XDto
XDetailDto

Entities are internal to the Data layer.

---

7. AUTOMAPPER

---

Use AutoMapper for mapping:

Entity ↔ DTO

Rules:

* All mappings must be centralized in Mapping Profiles.
* Controllers and Services should not manually map properties.

---

8. DEPENDENCY INJECTION

---

All dependencies must be registered using ASP.NET Core DI.

Rules:

* Use constructor injection only.
* Never use `new` to instantiate services or repositories.
* Follow correct lifetimes:

Scoped

* Repositories
* DbContext
* Business Services

Singleton

* Stateless infrastructure services

---

9. ASYNC PROGRAMMING

---

All database and I/O operations must use async/await.

Rules:

* Use async EF Core methods
* Never block threads with `.Result` or `.Wait()`

Example:

await _repository.GetByIdAsync(id);

---

10. VALIDATION

---

Validation must be implemented using FluentValidation.

Rules:

* DTO validation rules must be placed in `/Validators`.
* Controllers should rely on automatic model validation.
* Return proper validation errors.

---

11. SECURITY RULES

---

Authentication:

* JWT authentication must be implemented.

Authorization:

* Use role-based authorization attributes.

Password Security:

* Passwords must be securely hashed.
* Never store plain text passwords.

Input Security:

* Validate all external input.

API Security:

* Protect create/update/delete endpoints.

---

12. ERROR HANDLING

---

The project must include a global exception handling middleware.

Rules:

* Do not expose internal exceptions to clients.
* Return standardized error responses.

Example HTTP codes:

200 OK
201 Created
400 Bad Request
401 Unauthorized
403 Forbidden
404 Not Found
500 Internal Server Error

---

13. PAGINATION

---

All list endpoints must support pagination.

Standard parameters:

pageNumber
pageSize

Rules:

* Avoid returning large datasets
* Default page size must be reasonable

---

14. PERFORMANCE GUIDELINES

---

To maintain high performance:

* Use proper database indexes
* Avoid N+1 queries
* Use projection queries when possible
* Use caching where beneficial
* Keep queries efficient

---

15. CODE QUALITY

---

Code must follow these standards:

* Meaningful naming
* Small focused methods
* Clear separation of responsibility
* No duplicated logic
* Consistent formatting
* Proper comments for complex logic

---

16. PROHIBITED PRACTICES

---

The following are NOT allowed:

* Business logic inside Controllers
* Direct DbContext usage inside Controllers
* Returning Entities directly from APIs
* Hardcoded configuration values
* Blocking async calls
* Mixing responsibilities across layers

---

17. FUTURE SCALABILITY

---

The architecture must support future upgrades such as:

* Distributed caching
* Microservices
* Message queues
* Background workers
* Horizontal scaling

Design decisions must avoid tight coupling.

---

18. JS ARCHITECTURE RULES — FORUM WEBSITE

These rules govern all client-side JavaScript in this project.

1. ONE JS FILE PER VIEW
   • Each MVC view that needs JavaScript has exactly one dedicated file.
   • Location: wwwroot/js/pages/<view-name>.js  (kebab-case)
     Examples: home.js, post-detail.js, create-post.js, user-profile.js
   • The file contains an IIFE module + one DOMContentLoaded listener.
   • No page logic lives in _Layout.cshtml or any shared file.

2. INCLUDE JS VIA @section Scripts
   • Views include their page script through the MVC section mechanism:
       @section Scripts {
           <script src="~/js/pages/<page-name>.js" asp-append-version="true"></script>
       }
   • _Layout.cshtml renders the section just before </body>, after all shared scripts:
       @RenderSection("Scripts", required: false)
   • No inline <script> blocks and no onclick/on* attributes anywhere.
     (CSP: script-src 'self' — unsafe-inline is forbidden.)
   • For view-specific CSS (e.g. Quill Snow), use @section Styles which renders in <head>:
       @section Styles {
           <link rel="stylesheet" href="..." />
       }
     _Layout.cshtml renders it at the end of <head>:
       @RenderSection("Styles", required: false)

3. API CALLS LIVE IN wwwroot/Services/
   • No direct fetch() calls in page scripts or shared JS.
   • All HTTP communication goes through named service IIFEs:
       wwwroot/Services/ApiService.js      — base GET/POST/PUT/DELETE wrapper
       wwwroot/Services/PostService.js     — /api/post/* methods
       wwwroot/Services/UserService.js     — /api/user/* methods
       wwwroot/Services/CommentService.js  — /api/comment/* methods
   • Service files are pure (no side-effects on load) and are loaded
     globally in _Layout.cshtml after bootstrap.bundle.min.js.
   • Standard method names: getAll, getById, getByUser, create, update,
     remove, login, register, logout, me, getProfile, close.

4. SHARED UTILITIES
   • wwwroot/js/utils.js  — Utils IIFE: timeAgo, avatarHtml, flash,
                            slugify, escapeHtml, fmtNum, strToColor.
                            Also exports _quillToolbar() (shared by editor pages).
   • wwwroot/js/auth.js   — Auth IIFE: current-user state, nav update,
                            login/register/logout/search form wiring.
                            Owns the single global DOMContentLoaded that wires
                            all modals and resolves Auth.ready.
                            Exposes Auth.ready — a Promise resolved after
                            Auth.init() completes (user fetched from /api/user/me).
   • Both loaded globally in _Layout.cshtml, after all Service files.

5. AUTH ORDERING — await Auth.ready
   • Page scripts MUST await Auth.ready before accessing Auth.user
     or rendering any user-sensitive UI:
       document.addEventListener('DOMContentLoaded', async () => {
           await Auth.ready;
           await MyPage.init();
       });
   • This guarantees correct auth state regardless of network timing.
   • Never call Auth.init() from a page script — auth.js owns that.

6. QUILL EDITOR — LOAD ON DEMAND
   • Quill CSS and JS are NOT loaded globally in _Layout.cshtml.
   • Only views that use the Quill editor (PostDetail, CreatePost) include
     both the CSS and the script in their sections:
       @section Styles {
           <link rel="stylesheet" href="https://cdn.jsdelivr.net/npm/quill@1.3.7/dist/quill.snow.css" />
       }
       @section Scripts {
           <script src="https://cdn.jsdelivr.net/npm/quill@1.3.7/dist/quill.min.js"></script>
           <script src="~/js/pages/<page>.js" asp-append-version="true"></script>
       }
   • The page script comes AFTER the Quill CDN script so `new Quill(...)` is available.

7. NAMING & STYLE CONVENTIONS
   • Module IIFEs: PascalCase (Forum, PostDetail, CreatePost, UserProfile).
   • Private helpers inside IIFEs: _camelCase prefix (_load, _renderRow...).
   • Service methods: camelCase (getAll, create, remove...).
   • No global variables other than the named IIFEs and service objects.
   • 'use strict'; at the top of every file.

LOAD ORDER IN _Layout.cshtml (bottom of <body>)
   1. jquery.min.js                (Bootstrap dependency)
   2. bootstrap.bundle.min.js      (UI components)
   3. ApiService.js                (base HTTP)
   4. PostService.js               (uses ApiService)
   5. UserService.js               (uses ApiService)
   6. CommentService.js            (uses ApiService)
   7. utils.js                     (Utils + _quillToolbar)
   8. auth.js                      (Auth + global DOMContentLoaded)
   9. @RenderSection("Scripts", required: false)   <- page script goes here
