PHASE 1 FEATURES (MVP)
=====================================

1) User System
- Register (username, email, password)
- Login (generate JWT token)
- Secure password hashing
- Role system (User, Admin)
- Get user profile page
- User profile summary view (/profile/{id}):
  - Shows: username, role, joined date, total posts count
  - Lists all posts by that user (title, view count, comment count, created date)
  - Paginated post list (same pageSize as homepage)
  - Accessible by anyone (public view)
  - Post rows link to /post/{slug}/{id}

2) Post System
- Create post (title, content)
- Edit post (only owner) — NOT admin
- Delete post (owner or admin)
- Close/reopen post (admin only) — toggles IsClosed; shown as badge on post title
- Get post list (with pagination)
- Get post detail by id
- Post URL uses slug: /post/{slug}/{id} — slug computed from title client-side, id is authoritative
  - Slug auto-corrects via history.replaceState if manually changed

Button visibility rules (enforced in JS _renderPost):
- Edit button:   owner only
- Delete button: owner or admin
- Close button:  admin only (label toggles "Đóng" / "Mở lại")

3) Comment System
- Add comment to post
- Edit comment (only owner)
- Delete comment (owner or admin)
- Get comments by post

3a) Category System
- Every post must belong to exactly one category
- Default category "Uncategorized" seeded with IsDefault=true
- Seeded categories (in order): Uncategorized, General, Q&A, Tutorial, Announcement
- Admin CRUD: create, update, delete categories
  - Cannot delete the default category (IsDefault=true)
  - Cannot delete a category that still has posts (FK Restrict)
  - SortOrder field for display ordering
- Public: GET /api/categories (list), GET /api/categories/{id}
- Post.CategoryId is required; CreatePostDto.CategoryId=0 resolves to the default category

3b) Tag System
- One post can have zero to many tags (max 5 tags per post)
- Tags are simple Name-only entities (no slug) — unique name, case-insensitive
- Admin CRUD: create, update, delete tags
  - Deleting a tag cascades to PostTags join rows (posts keep the rest of their tags)
- Public: GET /api/tags (list), GET /api/tags/{id}
- Post owner assigns tags by ID when creating/editing a post
  - CreatePostDto.TagIds (optional list of existing tag IDs)
  - UpdatePostDto.TagIds: null = keep current tags; empty list = clear all tags
- EF Core many-to-many via implicit PostTags join table
- PostDto and PostDetailDto include List<TagDto> Tags and CategoryName

4) View Count System (Discourse-style)
- Count only "valid views", not every GET request
- A valid view must:
  - Come from a real browser (non-empty User-Agent, no bot patterns)
  - Not repeat from the same user/session within a configurable time window
- Implementation:
  - IViewCountService.ShouldCount(postId, viewerKey, userAgent) → bool
  - viewerKey: "u:{userId}" for authenticated, "g:{ip}" for guests
  - Backed by IMemoryCache — one cache entry per (postId, viewerKey), TTL = window
  - On ShouldCount=true: call PostService.IncrementViewCountAsync (atomic SQL UPDATE)
  - Reflect new ViewCount in response DTO (post.ViewCount++) — no extra DB round-trip
- Bot detection (User-Agent substring match, case-insensitive):
  - Blocked: bot, crawl, spider, slurp, fetch, scan, python, curl, wget,
             axios, java/, go-http, headless, phantom, selenium, puppeteer
  - Empty/missing User-Agent → treated as bot
- Config: appsettings.json "ViewCount": { "WindowMinutes": 10 }  (clamped 1–60)
- No Redis required for Phase 1 (IMemoryCache is sufficient)
- No per-user DB deduplication table needed for Phase 1
- ViewCountService registered as Singleton (cache lives for the process lifetime)
- PostService.GetPostByIdAsync no longer auto-increments;
  increment only happens when PostController decides ShouldCount=true
- Phase 2 upgrade path: swap IMemoryCache for IDistributedCache (Redis)
  and add a batch-flush BackgroundService for write aggregation

5) Responsive UI
- All views must be fully responsive across:
  - Mobile: 320px–480px (iPhone SE, iPhone 14, Samsung Galaxy)
  - Tablet: 768px–1024px (iPad, iPad Pro)
  - Desktop / Laptop: 1280px and above
- Use Bootstrap 5 breakpoints (sm / md / lg / xl)
- No horizontal scroll on any breakpoint
- Touch-friendly tap targets (min 44px)
- Navigation collapses to hamburger menu on mobile
- Topic list columns (replies, views, activity) hidden on small screens (d-none d-md-block)
- Post detail and comment forms stack vertically on mobile
- Modals (login, register, edit post) must be usable on small screens

FEATURE 6 — ADMIN DASHBOARD (added)
=====================================

Route: GET /admin  (requires Admin role — enforced both server [Authorize] and client-side redirect)

Bootstrap 5 tabbed single-page admin panel with four lazy-loaded tabs:

6a) Category Management
- Tab: Danh mục — lists all categories with Id, Name, Description, IsDefault, SortOrder, PostCount
- Create: POST /api/category { name, description, sortOrder, isDefault }
- Update: PUT  /api/category/{id}
- Delete: DELETE /api/category/{id}
  - Delete button disabled if IsDefault=true OR PostCount > 0 (FK Restrict prevents accidental orphan)

6b) Tag Management
- Tab: Thẻ — lists all tags with Id, Name, PostCount
- Create: POST /api/tag { name }
- Update: PUT  /api/tag/{id}
- Delete: DELETE /api/tag/{id} (cascades PostTags automatically)

6c) Post Moderation
- Tab: Bài viết — lists all posts paginated (20/page) with Id, Title, Author, Category, Status
- Close/Reopen: PUT /api/post/{id}/close  (toggles IsClosed)
- Delete: DELETE /api/post/{id}
- Pagination via Utils.renderPagination()

6d) User Management
- Tab: Người dùng — lists all users paginated (20/page) with Id, Username, Email, Role, IsActive
- Toggle Active: PUT /api/user/admin/{id}/toggle-active  (ban / unban)
  - Throws BusinessRuleException if admin targets their own account
- Pagination via Utils.renderPagination()

New API endpoints added:
  GET /api/user/admin/users?page=N&pageSize=N   — admin only, returns PagedResult<AdminUserDto>
  PUT /api/user/admin/{id}/toggle-active         — admin only, returns 200 with no data body

New DTO:
  AdminUserDto — { Id, Username, Email, Role, IsActive, CreatedAt } — manual projection (no AutoMapper)

New MVC controller:
  AdminController (GET /admin) — class-level [Authorize(Roles=Admin)], returns View()

Navigation:
  ddAdminLink <li> in _Layout.cshtml dropdown — hidden (d-none) by default,
  revealed by auth.js _updateNav() when user.role === 'Admin'
